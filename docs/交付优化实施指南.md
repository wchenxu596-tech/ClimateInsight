# ClimateInsight 项目交付优化实施指南

> 目标：将当前“可展示的原型”整理为一个**可在新环境中复现、可验证数据正确性、可部署演示、可交付答辩**的 ClimateInsight 全球气候智能分析项目。
>
> 本文以当前仓库实际代码为准。实施期间请先创建独立分支；所有密码、容器名称和本机路径都不得继续硬编码。

---

## 1. 当前问题与改造原则

当前目标链路应统一为：

```text
NOAA GSOD 原始 CSV
  -> Python 预处理与质量校验
  -> Hive（ODS / DWD，可追溯明细）
  -> Python 聚合
  -> MySQL（面向 API 的 DWS / ADS）
  -> Flask REST API
  -> Vue 可视化前端
```

当前仓库中同时存在 Hive DWS/ADS 脚本和 Python->MySQL 聚合脚本，两套口径已经漂移。为保证交付，采用以下原则：

1. **Hive 只保存原始及清洗明细（ODS、DWD）**，MySQL 保存 API 所需的月度汇总与看板指标。
2. **Python ETL 作为唯一的 DWD->MySQL 聚合实现**，避免维护两套 DWS/ADS SQL。
3. 所有服务通过环境变量配置；Docker Compose 是唯一推荐启动方式。
4. 每个阶段必须有可重复执行的验证命令和明确验收条件。
5. 当前只包含 2024 年数据时，产品文案必须写“2024 年全球气候分析”，不得声称已完成 140 年趋势分析。

---

## 2. 推荐交付范围

### 2.1 必须交付（MVP）

- 2024 年 NOAA GSOD 数据的可复现导入、清洗、聚合流程。
- 可一键启动的 MySQL、后端、前端环境。
- 总览、月度趋势、站点排名、气候带分布四类真实数据看板。
- 健康检查、统一错误响应、接口文档和部署说明。
- 数据质量报告、测试结果和演示脚本。

### 2.2 可选交付（时间充足后）

- 导入 2015–2023 年数据，才开放跨年变化趋势。
- 真正的 NL2SQL：只读数据库账号、SQL 白名单、SQL 校验、限时/限行数。
- 自动生成气候分析报告、世界地图和定时更新。

### 2.3 本轮不应承诺

- 没有多年份数据时的“百年变暖趋势”。
- 未接入大模型时的“AI 智能问答”。
- 没有 Agent 代码、调度与输出物时的“智能报告 Agent”。

### 3.3 移除硬编码配置


修改 `backend/app.py` 和 ETL 脚本：

- 禁止出现 `root/123456`。
- Flask 调试模式只允许在本地开发环境开启。
- MySQL 使用权限最小化的 `climate_app` 账号；NL2SQL 若实施，必须另建只读账号。
- 容器名不可写死；服务通过 Compose 服务名连接，例如 `mysql`、`hive-server`。

### 3.4 编写 Docker Compose

最低限度先编排三个服务：`mysql`、`backend`、`frontend`。Hive 若已由课程公共环境提供，可在运行手册中声明为外部依赖；若要完全交付，必须再补充 Hive 服务或明确导出后的数据导入方案。

Compose 要求：

- MySQL 使用命名卷，初始化 SQL 挂载至 `/docker-entrypoint-initdb.d`。
- 后端依赖 MySQL 健康检查，而非仅依赖启动顺序。
- 前端构建产物由 Nginx 提供，并将 `/api` 代理给后端。
- 仅暴露前端端口；开发时可额外映射 MySQL 端口。
- 加入健康检查、重启策略和 UTF-8 时区设置。

### 3.5 阶段验收

```powershell
Copy-Item .env.example .env
docker compose config
docker compose up -d --build
docker compose ps
```

验收：服务状态均为运行；MySQL 健康检查通过；未在版本库中出现真实密码或构建产物。

## 4. 阶段二：修复并统一数据模型

### 4.1 明确数据层职责

| 层 | 存储 | 作用 | 是否对 API 直接开放 |
| --- | --- | --- | --- |
| ODS | Hive | 原始 NOAA CSV，保留来源与装载批次 | 否 |
| DWD | Hive | 清洗、单位转换、质量标记后的日明细 | 否 |
| DWS | MySQL | 气象站-年月粒度汇总 | 否 |
| ADS | MySQL | KPI、月度趋势、排名、气候带 | 是 |

### 4.2 修复 CSV 读取与 ODS

原始 NOAA 文件的 `NAME` 字段包含逗号，因此不能使用普通的 `FIELDS TERMINATED BY ','` 解析。

实施步骤：

1. 在 Hive ODS 中使用 `OpenCSVSerde`，或在进入 Hive 前统一使用 Python 的 `csv.DictReader` 转换为 Parquet/标准 CSV。
2. ODS 的字段必须与原始 28 列头严格对应；为每次导入记录 `source_file`、`load_batch_id`、`ingested_at`。
3. 任选一种格式作为唯一事实来源，避免同时读取原始 CSV 与另一种列定义的 `climate_2024.csv`。
4. 在导入后比对：文件数、表头、记录数、站点数和日期范围。

建议的检查指标：

```text
2024 文件数：12,160
理论最大日记录数：12,160 × 366 = 4,450,560
检查项：导入行数不能远超理论上限；站名字段不得拆列。
```

### 4.3 修复 DWD 数据质量规则

在 `sql/02_dwd_etl.sql` 或统一的 Python 清洗模块中实现以下规则，并写入 `quality_flag` 与 `reject_reason`：

- NOAA 缺失哨兵值：温度、露点、气压、能见度、降水、雪深分别按数据字典处理，不能只过滤 `temp_mean`。
- 转换单位前先过滤无效值，转换后校验合理区间：温度建议 `[-90, 60] °C`，降水应为非负且设置业务上限。
- 经纬度 0 是合法值，不能作为过滤条件；仅过滤空值、无法解析值或超出经纬度范围的值。
- 日期必须属于目标年份；站点 ID、日期应构成唯一键。
- `FRSHTT` 必须规范化为 6 位 `0/1` 标记。
- 每批生成：输入行数、有效行数、拒绝行数、各拒绝原因计数。

### 4.4 统一 MySQL 表设计

删除“靠脚本运行时建表”的临时策略，改为版本化 SQL 迁移文件，例如：

```text
sql/mysql/001_schema.sql
sql/mysql/002_indexes.sql
sql/mysql/003_seed.sql
```

`dws_station_monthly` 至少应包含：

```text
station_id, station_name, year, obs_month,
latitude, longitude,
avg_temp, avg_temp_max, avg_temp_min,
max_temp, min_temp, total_precip,
rainy_days, extreme_days, obs_days, climate_zone,
PRIMARY KEY (station_id, year, obs_month)
```

建立索引：

- `(year, obs_month)`：月度趋势。
- `(year, climate_zone)`：气候带分布。
- `(year, avg_temp_max)`、`(year, total_precip)`：排名。

ADS 表应包含 `data_year` 或 `snapshot_date`，确保看板清楚说明数据范围。

### 4.5 清理失效 SQL

在 `sql/04_ads_etl.sql` 中处理以下问题：

- 删除或补齐 `data_completeness`、`latitude`、`longitude` 等引用字段。
- 统一 ODS 表名，不能同时出现 `ods_climate_raw` 与 `ods_climate_daily`。
- DWS/ADS 如果不再由 Hive 执行，应把该脚本标记为历史方案，或移入 `docs/archive/`，避免重复运行。
- 所有排名 SQL 必须以 `year` 为条件；没有多年数据时固定为 2024。

### 4.6 阶段验收

新增 `scripts/verify_data.py`，至少执行：

```text
1. CSV 文件数与站点数检查
2. ODS / DWD / DWS 行数检查
3. DWD 主键重复检查
4. 温度、降水越界检查
5. 温度、降水缺失哨兵残留检查
6. 站点和日期范围检查
7. KPI 与明细聚合结果交叉校验
```

验收：没有语法错误；没有无法解析的 CSV；最高温、最低温、降水都处于定义的合理区间；校验报告可存档。

## 5. 阶段三：重构并验证 ETL

### 5.1 修复当前阻断问题

`scripts/etl_to_mysql.py` 当前存在 `else` 后再次出现 `elif` 的语法错误，必须先删除重复分支，再运行任何 ETL。

随后将该文件拆分为职责单一的模块：

```text
scripts/
  extract_hive.py       # 流式读取 DWD
  transform_metrics.py  # 聚合与气候带规则
  load_mysql.py         # 批量 Upsert
  verify_data.py        # 质量校验
  run_etl.py            # 流程入口
```

### 5.2 避免内存和覆盖问题

实施要求：

- 禁止 `rows = list(reader)` 读取全部 DWD 行；使用迭代器、分块查询或临时 Parquet/CSV 分区处理。
- 聚合键必须为 `(station_id, year, obs_month)`。
- 使用 `INSERT ... ON DUPLICATE KEY UPDATE` 或按 `load_batch_id` 进行原子替换，禁止无条件 `TRUNCATE` 作为唯一保障。
- 对数据库连接、Beeline 调用、Docker 调用设置超时、错误输出和重试策略。
- 仅在所有聚合数据验证通过后再更新 ADS 快照。

### 5.3 统一气候带口径

将气候带判断写成一个独立函数，并由单元测试覆盖。明确它是“演示型规则”，不是严格的柯本气候分类。

建议优先级示例：

```text
polar        : 绝对纬度 >= 60 或年均温 < 0
tropical     : 年均温 >= 18 且月最低均温 >= 18
temperate    : 年均温 [10, 18)
arid         : 年均降水低于阈值（阈值需在文档注明）
continental  : 其他情况
```

### 5.4 ETL 验收

```powershell
python -m compileall -q backend scripts
python scripts/run_etl.py --year 2024
python scripts/verify_data.py --year 2024
```

验收：ETL 可重复运行；第二次运行不产生重复记录；MySQL 的 KPI、排名、月趋势和气候带表均有数据且与 DWS 交叉校验一致。

## 6. 阶段四：完善 Flask API

### 6.1 重构目录

建议按如下结构拆分：

```text
backend/
  app.py
  config.py
  db.py
  routes/
    health.py
    dashboard.py
    rankings.py
    analytics.py
    ai.py
  schemas.py
  requirements.txt
```

### 6.2 接口契约

统一响应格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "meta": { "data_year": 2024 }
}
```

实现并记录下列接口：

| 接口 | 关键参数 | 要求 |
| --- | --- | --- |
| `GET /api/health` | 无 | 返回服务与数据库状态 |
| `GET /api/kpi` | `year` | 返回 KPI 和数据年份 |
| `GET /api/monthly` | `year` | 返回 1–12 月均温、均最高/最低温 |
| `GET /api/zones` | `year` | 返回气候带及站点数 |
| `GET /api/ranking` | `year, category, limit` | category 白名单，limit 限制为 1–50 |
| `GET /api/trend` | `start_year, end_year` | 仅有一年时返回单年/月度趋势说明，不伪造同比 |

### 6.3 后端安全与稳定性

- 使用参数化 SQL；枚举参数（如排名类别）仍需白名单校验。
- `limit`、年份等参数必须验证类型与范围，错误返回 400 而非 500。
- 统一捕获数据库异常，返回不泄露内部细节的 500 响应，同时记录结构化日志。
- 使用连接池或 Flask 应用上下文管理连接。
- CORS 仅允许指定前端来源；生产通过 Nginx 同源代理时可关闭跨域。
- 生产启动使用 Gunicorn/Waitress，不使用 Flask 自带 `debug=True` 服务器。

### 6.4 处理 AI 功能

在未实现前，把页面改名为“分析助手（建设中）”，或隐藏入口。若实施 NL2SQL，必须按下列顺序完成：

1. 给模型提供仅包含允许表、字段和示例查询的 Schema。
2. 生成 SQL 后进行 AST 或严格规则校验：只允许单条 `SELECT`，禁止 DDL/DML、子查询深度过大和跨库访问。
3. 强制加 `LIMIT`、执行超时和最大扫描量。
4. 用只读 MySQL 用户执行。
5. 记录问题、生成 SQL、耗时、行数和错误，不记录敏感信息。
6. 先用固定问题集回归测试，再开放自由提问。

### 6.5 API 验收

为每个接口写 pytest 测试，并增加 Postman/Bruno 集合。至少验证：成功、空数据、非法参数、数据库不可用四种情况。

```powershell
pytest -q
curl http://localhost:5000/api/health
curl "http://localhost:5000/api/ranking?year=2024&category=hottest&limit=10"
```

## 7. 阶段五：完善 Vue 前端

### 7.1 配置化 API

将固定的 `http://localhost:5000/api` 改为：

```javascript
const api = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || '/api',
  timeout: 15000,
})
```

开发环境通过 Vite proxy 转发 `/api`，生产环境由 Nginx 转发，避免浏览器跨域和地址写死。

### 7.2 修正数据契约

- 趋势页不要读取后端未返回的 `temp_change`。只有多年度数据具备后，再展示同比变化。
- 图表标题、单位、Tooltip 和数据年份必须来自 API `meta`，不应固定写 2024。
- 排名图需根据类别切换单位（°C、mm、天数）。
- 气候带图说明分类方法是简化规则。

### 7.3 补齐体验与可访问性

每个页面统一实现：

- 加载中的骨架屏或 `el-skeleton`。
- 接口失败的可见错误提示和“重试”按钮。
- 空数据状态。
- 移动端布局：KPI 卡片、导航和图表在窄屏可换行。
- 图表数据为空时不初始化误导性坐标轴。

移除默认 Vite 残留文件、未使用的 `HelloWorld.vue`、默认素材和模板 README，替换为项目实际说明。

### 7.4 前端验收

```powershell
cd frontend
npm ci
npm run build
```

验收：生产构建无错误；断开后端时页面给出明确提示；四个数据页面可展示真实 API 数据；浏览器控制台无未处理异常。

## 9. 最终交付清单

提交前逐项核对：

- [ ] `docker compose up -d --build` 在新环境可启动。
- [ ] `.env.example` 完整，真实 `.env` 未提交。
- [ ] 后端和 ETL 无 Python 语法错误，依赖有锁定或明确版本。
- [ ] 2024 年 ETL 可重跑，结果不重复、不覆盖其他年份。
- [ ] 数据质量报告通过，异常温度/降水已隔离。
- [ ] API 文档、健康检查、错误响应均可验证。
- [ ] 前端生产构建成功，四个核心页面显示真实数据。
- [ ] AI 功能已实现并受控，或明确标为未交付。
- [ ] README、数据字典、部署手册和演示脚本已更新。
- [ ] 已保留测试报告、接口样例和关键页面截图。

满足以上清单后，项目可以作为“2024 年 NOAA 气候数据驱动的 BI 分析系统”直接交付；多年份趋势、地图和 NL2SQL 可作为后续增值迭代，而不影响 MVP 的完整性与可信度。
