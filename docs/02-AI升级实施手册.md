# ClimateInsight AI 分析助手升级实施操作手册

> 适用对象：负责在现有 ClimateInsight 项目中实施升级的 AI 或开发者。  
> 目标：将现有“单年/有限多年 GSOD 问答助手”升级为可扩展的全历史气候分析平台，同时保留原有问答能力和安全边界。  
> 工作原则：先盘点、再设计、后实现；每一步均须通过验收后才进入下一步。除明确标出的迁移步骤外，不删除旧功能或旧表。

---

## 0. 实施边界与最终交付物

### 0.1 必须实现的能力

1. 从自然语言中提取时间、空间、气候要素、统计口径和一个或多个分析意图。
2. 查询 GSOD 与新增 GHCN-Daily 数据，支持日、月、年粒度及任意年份范围；未来年份无需改代码即可接入。
3. 提供趋势、异常、对比、周期/季节分析，返回可解释的统计指标与结论。
4. 依据分析结果自动选择 ECharts 图表，支持图表—明细表联动。
5. 支持预警规则、建议生成、报告异步生成与下载。
6. 支持知识库检索增强、多轮对话、分析链路展示、用户权限控制和观测指标。
7. 新增“高级分析、预警中心、知识库管理、报告中心”前端入口，并保留原 AIPanel 的快捷问答体验。

### 0.2 交付物清单

完成后必须同时提交：

- 后端升级代码、数据库迁移脚本、初始化/导入脚本与环境变量示例；
- 前端页面、组件、接口封装与路由；
- OpenAPI 或等价接口文档；
- 管理员/用户使用手册；
- 自动化测试、测试数据和测试报告；
- 部署说明、回滚方案、监控告警说明。

### 0.3 不可违反的约束

- 所有数据库查询必须使用参数化占位符；禁止把 LLM 生成文本直接拼入 SQL。
- LLM 只能输出受 JSON Schema 约束的“分析计划”，不能拥有任意 SQL 或任意工具调用权限。
- 用户的年份范围、空间范围、字段名、排序方式、分页参数均须白名单校验。
- 原 `POST /api/agent/query` 必须兼容，已有 `kpi/monthly/ranking/zones/...` 意图不可回归。
- 外部文档、知识库内容和用户问题均是不可信输入，必须防提示词注入、越权检索和跨租户泄露。

## 2. 第二阶段：目标架构与代码边界

### 2.1 推荐的调用链

```text
用户问题 / 高级分析表单
  → API 鉴权与输入校验
  → 会话上下文 + 知识库检索（按用户/权限过滤）
  → NLU：规则预解析 + LLM Function Calling + Schema 校验
  → AnalysisPlan（结构化分析计划）
  → 权限裁剪、成本/超时评估、工具编排
  → 数据查询工具 → 统计工具 → 预警工具 → 图表规格工具
  → 同步结果或异步报告任务
  → 结果存档、可选知识入库（人工审核/策略控制）
  → 前端渲染文字、图表、表格、预警和分析链路
```

### 2.2 建议模块划分

在已有 `agent` 包内扩展或等价映射为下列职责。若项目已有同类目录，应合并而非重复实现。

```text
agent/
  schemas.py          # Pydantic/JSON Schema：请求、计划、结果、预警
  planner.py          # NLU 结果归一化、LLM function calling、规则回退
  catalog.py          # 数据集、字段、聚合、空间过滤白名单
  query_builder.py    # 仅接受白名单 AST/QuerySpec，生成参数化 SQL
  tools/
    climate_data.py   # GSOD/GHCN 查询与分页/采样
    statistics.py     # 趋势、异常、检验、周期工具
    alerts.py         # 规则评估与建议匹配
    charts.py         # 数据→ChartSpec 的确定性映射
    reports.py        # 报告任务创建、状态、导出
    knowledge.py      # RAG 检索、入库、审核
  orchestration.py    # 工具调用顺序、超时、降级、审计
  responder.py        # 兼容旧回复并生成新结构化回复
```

### 2.3 关键实现决策

- `AnalysisPlan` 是后端唯一允许跨模块传递的分析请求结构；严禁把自然语言或 LLM 原始输出直接传给 SQL 工具。
- 轻量查询（预计少量聚合、<2 秒）同步返回；报告和大范围日值查询提交 Celery 任务，返回 `job_id`。
- 所有统计函数返回“数值 + 方法 + 样本数 + 有效数据范围 + 限制说明”，避免只生成结论。
- 图表配置由规则工具生成；LLM 只可建议 `chart_goal`，不可生成可执行 JavaScript。
- 用功能开关 `ADVANCED_ANALYSIS_ENABLED` 逐步发布；旧 `/agent/query` 保持可用。

## 3. 第三阶段：数据层、迁移与性能

### 3.1 建立数据模型

保留现有 ADS/DWS 表，并新增或映射以下逻辑实体。实际字段和类型须按所用数据库调整，所有表都应含数据版本、创建/更新时间和来源标识。

| 逻辑表 | 最小关键字段 | 目的 |
| --- | --- | --- |
| `dim_station` | `station_id, name, country, admin_area, latitude, longitude, elevation, active_from, active_to` | GSOD/GHCN 统一站点维度 |
| `ods_ghcn_daily` | `station_id, obs_date, element, value, unit, quality_flag, source` | GHCN-Daily 原始日值 |
| `dws_climate_daily` | `station_id, obs_date, tavg, tmax, tmin, prcp, wind_speed, pressure, quality_flags` | 规范化日粒度宽表/视图 |
| `ads_climate_monthly` | `station_id/region_id, data_month, element, avg_value, min_value, max_value, sum_value, sample_count` | 月聚合 |
| `ads_climate_annual` | `station_id/region_id, data_year, element, ...` | 年聚合 |
| `ads_climate_percentiles` | `scope_type, scope_id, element, season, baseline_start, baseline_end, p05, p20, p95, mean, stddev` | 异常/预警基线 |
| `alert_rule` | `id, tenant_id, name, scope, element, operator, threshold_kind, threshold_value, severity, enabled` | 内置与自定义规则 |
| `alert_event` | `id, rule_id, result_ref, triggered_at, severity, status, evidence_json` | 可追溯预警 |
| `analysis_run` | `id, user_id, session_id, plan_json, status, data_version, timing_json` | 分析链路和审计 |
| `report_job` | `id, analysis_run_id, format, status, file_key, expires_at, error` | 异步报告 |
| `knowledge_document/chunk` | `id, tenant_id, source_type, content, metadata_json, embedding, status` | 知识与向量索引 |
| `conversation_message` | `session_id, role, content, analysis_run_id, created_at` | 多轮会话 |

### 3.2 迁移执行顺序

1. 编写可重复执行的数据库迁移：先维度表、原始表，再聚合表、业务表、索引和权限策略。
2. 为时间字段建立分区（建议按年或月，取决于数据量）；为 `station_id + obs_date`、`element + obs_date`、地理字段和常用聚合维度创建索引。
3. 导入 GHCN-Daily 前先建立数据质量规则：日期合法、站点存在、元素单位统一、重复记录策略、缺失值和质量标记处理。
4. 导入采用可恢复批处理：每批记录水位线、来源文件校验和、失败行清单；禁止一次性无记录全表导入。
5. 生成日→月→年聚合和百分位基线；将聚合结果与抽样原始日值人工核对。
6. 对高频组合（区域+年份+要素+季节）建立物化视图或 Redis 缓存，并设定失效条件：数据版本变更、规则变更、TTL 到期。

### 3.3 数据验证门禁

- 同一站点、同一日、同一要素不能出现未解释的重复值；
- 单位统一：温度、降水、风速、气压必须在数据字典中写明转换规则；
- 新聚合表与抽样原表的均值/求和误差须在约定容差内；
- 新旧 GSOD 指标在重叠维度上对比并说明口径差异；
- 每次导入产出数据质量报告并存入 `analysis_run` 或独立审计表。

## 4. 第四阶段：NLU、分析计划与安全 SQL

### 4.1 定义唯一的分析计划契约

使用 Pydantic（或当前后端等价验证库）定义并强制校验以下结构。以下为逻辑字段，枚举值必须在代码常量中统一维护。

```json
{
  "intent": ["compare", "trend", "anomaly"],
  "time_groups": [
    {"start": 2015, "end": 2020, "label": "基准期"},
    {"start": 2021, "end": 2025, "label": "对比期"}
  ],
  "granularity": "year",
  "season": [6, 7, 8],
  "spatial_filter": {
    "type": "region|station|climate_zone|bbox",
    "value": "..."
  },
  "metrics": ["avg_temperature"],
  "aggregation": "mean",
  "group_by": ["year"],
  "statistics": ["linear_regression", "mann_kendall", "zscore_2sigma"],
  "chart_goal": "trend",
  "page": "advanced_analysis",
  "confidence": 0.0
}
```

### 4.2 NLU 实施步骤

1. 先用规则预解析年份、月份/季节、站点编号、经纬度、明确的要素词和旧意图，快速覆盖确定性请求。
2. 为 DeepSeek 定义 Function Calling / JSON Schema，只允许返回 `AnalysisPlan`；系统提示中仅提供允许的要素、聚合、空间过滤和示例，不提供数据库凭据或原始 SQL。
3. 对 LLM 输出进行严格 Schema 验证、枚举校验、范围校验和语义校验（如结束年份不得早于开始年份）。失败、超时、置信度低时回退规则引擎或请求用户澄清。
4. 实现相对时间解析：以“当前数据最大年份”而非系统年份计算“近五年/这十年”；“所有年份”取数据目录中可用范围。
5. 实现复合请求拆解：一个问题可生成多个可组合子任务，但最大子任务数、时间跨度、站点数、返回行数均需上限。
6. 把最终计划、原始输入、规则/LLM 来源、置信度和修正记录入审计日志；日志脱敏，不记录密钥。

### 4.3 安全查询构建

1. 创建 `DatasetCatalog`，定义可查询数据集、字段映射、可用聚合、可 JOIN 关系、过滤器、最大范围和默认排序。
2. `query_builder` 只接收已验证的 `QuerySpec/AnalysisPlan`；SQL 的表名和列名仅能从 Catalog 选择，值一律作为数据库参数绑定。
3. 对空间过滤使用预定义实现：站点 ID 集合、行政区映射、气候带枚举或参数化 bbox；禁止接收任意 SQL 片段或函数名。
4. 设置查询超时、最大扫描分区、最大返回行数和分页令牌；超限时自动转为采样、聚合或异步任务，并向用户说明。
5. 用数据库只读账号执行分析查询；数据库层启用行级权限或在 QuerySpec 中强制注入 `tenant_id/data_scope`。

### 4.4 NLU 验收用例

至少覆盖：

- “2015-2020年与2021-2025年中国东部夏季平均温度变化，并识别异常年份”；
- “近五年温带降水趋势”；
- “所有年份最热的十个站点”；
- “经度100到125、纬度20到45的2024年暴雨日数”；
- “把刚才图表改为对数坐标”（上下文修正，不改变数据范围）；
- 无效年份、倒置年份、未知要素、超大 bbox、SQL 注入文本和提示词注入文本。

## 5. 第五阶段：统计、预警、知识库与报告服务

### 5.1 统计工具

按独立纯函数实现，并在输出中写明前提与失败原因。

| 工具 | 输入 | 输出最低要求 |
| --- | --- | --- |
| 线性趋势 | 时间序列 | 斜率、截距、R²、p 值、样本数、单位/十年 |
| Mann-Kendall | 时间序列 | S、tau、p 值、显著性级别、趋势方向 |
| 滑动平均 | 序列+窗口 | 平滑序列、窗口、边界处理规则 |
| Z-score 异常 | 序列+基线 | 均值、标准差、z 值、阈值、异常标记 |
| 极端事件 | 日值+规则 | 事件起止、持续天数、阈值来源、涉及站点 |
| t 检验/ANOVA | 两组/多组样本 | 均值差、统计量、p 值、样本量、适用性提示 |
| 季节/周期 | 足够长序列 | 季节项/周期项及方法限制；数据不足则拒绝推断 |

统计库与版本要固定；对缺失值、异常值、样本量不足和多重比较给出明确处理规则。未经验证不得把相关性写成因果关系。

### 5.2 预警规则与建议

1. 预置规则：温度高于历史 P95、降水低于 P20、连续高温 ≥5 天、连续少雨 ≥3 个月；每条规则含基线范围、空间范围、严重度和适用要素。
2. 提供管理员管理接口：创建、禁用、版本化、模拟运行、审批和审计；普通用户只能创建自身有权限范围内的规则。
3. 分析完成后用规则引擎评估结果，生成 `alert_event`，其证据必须包括计算值、阈值、基线、数据版本与关联分析 ID。
4. 建议生成采用“预警类型 × 行业 × 严重度”的受审内容模板优先；LLM 可润色但不能虚构政策、规范或事实。引用知识库内容时附来源和可信度。
5. 预警触发应幂等：相同规则、同一数据版本、同一范围重复运行不得产生无穷重复通知。

### 5.3 知识库（RAG）

1. 选择并部署 Milvus（或获项目批准的向量库）；知识原文、元数据、向量、文档版本和审核状态必须可追溯。
2. 文档导入流程：病毒/格式检查 → 文本提取 → 分块 → 元数据标注（租户、来源、主题、权限、有效期）→ 嵌入 → 人工审核/发布。
3. 检索时先按租户、权限、状态、有效期过滤，再做向量和关键词混合检索；返回有限条数与来源元数据。
4. 只有通过审核的“稳定结论/规则/政策文档”可作为回答依据；用户即时分析结果默认只存历史，不自动写入全局知识库。
5. 实现删除/失效时的原文、向量、缓存和引用一致性清理；保留审计记录。

### 5.4 异步报告与导出

1. 配置 Celery worker、消息代理和结果存储；定义队列：`report`、`data_export`、`knowledge_ingest`，设置超时、重试、死信与幂等键。
2. 报告任务读取已固化的 `analysis_run` 快照，生成 Markdown/HTML，再生成 PDF；不得在报告生成期间重新执行不受控查询。
3. 报告最低章节：摘要、范围与口径、数据质量、关键发现、图表、统计表、预警、建议、方法限制、数据来源与生成时间。
4. 图表导出 PNG/SVG，数据导出 CSV/XLSX；下载文件放入受控对象存储，使用带权限和过期时间的下载地址。
5. 前端轮询或使用 WebSocket/SSE 显示进度、失败原因和可重试状态；不得让浏览器长时间同步阻塞。

## 6. 第六阶段：接口契约与前端实施

### 6.1 建议 API

保留旧接口，并新增版本化接口（路径可与现有 API 规范一致）：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| POST | `/api/agent/query` | 兼容旧快捷问答；内部转换为 AnalysisPlan |
| POST | `/api/v2/analysis/plan` | 仅解析/预览结构化计划，供用户确认 |
| POST | `/api/v2/analysis/runs` | 执行高级分析，返回结果或 `job_id` |
| GET | `/api/v2/analysis/runs/{id}` | 查询分析、链路、表格和图表规格 |
| POST | `/api/v2/analysis/runs/{id}/follow-ups` | 多轮追问或图表修正 |
| CRUD | `/api/v2/alert-rules` | 预警规则管理 |
| GET | `/api/v2/alerts` | 预警中心列表/筛选/确认状态 |
| POST/GET | `/api/v2/reports`、`/api/v2/reports/{id}` | 报告任务与状态 |
| CRUD | `/api/v2/knowledge/documents` | 知识导入、审核、检索和失效 |

### 6.2 统一响应模型

新接口应至少返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "run_id": "...",
    "status": "completed|queued|running|failed",
    "plan": {},
    "summary": "...",
    "tables": [{"id": "annual-series", "columns": [], "rows": [], "page": {}}],
    "charts": [{"id": "trend", "spec": {}, "drilldown": {}}],
    "statistics": [],
    "alerts": [],
    "suggestions": [],
    "lineage": [],
    "limitations": [],
    "meta": {"data_version": "...", "timing_ms": {}}
  }
}
```

`chart.spec` 使用受控的 ECharts Option 子集或项目定义的 `ChartSpec`；前端不得执行后端返回的任意脚本。

### 6.3 前端实施步骤

1. 抽取原 `AIPanel.vue` 的消息、表格、图表和错误提示能力为可复用组件，保证旧面板仍正常工作。
2. 新增“高级分析”页：自然语言输入、可编辑的解析条件（时间/空间/要素/统计/图表）、计划预览、运行按钮、结果区与分析链路。
3. 新增“预警中心”：严重度、时间、区域、要素、状态筛选；展示证据、建议、关联分析与确认/关闭操作。
4. 新增“知识库管理”：上传/导入状态、审核、标签、权限、过期/删除操作；该页仅管理员或被授权用户可见。
5. 新增“报告中心”：创建报告、模板/格式选择、异步进度、历史列表、权限下载、失败重试。
6. 实现图表选择渲染器：趋势→线/面积，比较→柱/条，分布→直方/箱线，占比→类别≤5 时饼/环，相关→散点，空间→地图。图表的色盲友好配色、空状态、移动端缩放和下载必须统一。
7. 实现图表—表格联动：点击数据点仅向后端发送受限的 `drilldown` 筛选键，后端再查询明细；不可把整批原始数据预先塞入浏览器。
8. 所有请求处理加载、取消、超时、权限不足、异步任务失败和空数据状态；大文件下载显示进度。

## 7. 第七阶段：测试、性能、安全与上线

### 7.1 自动化测试最小集

- 单元测试：时间解析、空间过滤、Schema 验证、QueryBuilder 白名单、每种统计函数、规则阈值、图表选择。
- 集成测试：GSOD/GHCN 查询、聚合、缓存失效、Celery 任务、向量检索权限过滤、报告导出。
- 回归测试：旧 17 意图、旧 API 响应字段、前端快捷问题、LLM 超时后的规则回退。
- 端到端测试：需求示例的“两个时期 + 中国东部 + 夏季温度 + 异常”完整流程、预警、报告下载和会话追问。
- 安全测试：SQL/NoSQL 注入、提示词注入、越权站点范围、跨用户会话、下载链接猜测、恶意文档上传、超大范围查询。
- 负载测试：简单查询 <2 秒、复杂分析 <10 秒、报告 <30 秒异步提交；至少 50 并发用户，记录 P50/P95、错误率和资源占用。

### 7.2 可观测性

对每个 `run_id` 记录并关联：用户/租户（脱敏）、解析来源与耗时、查询模板和耗时、扫描行数、缓存命中、统计耗时、图表耗时、LLM 调用、规则触发、任务状态和失败栈。建立如下告警：

- API 5xx、任务失败率、队列积压；
- 慢查询、缓存失效率、LLM 超时/Schema 失败率；
- 数据导入质量异常、预警规则异常激增；
- 向量库不可用、对象存储下载失败、权限拒绝异常增多。

### 7.3 发布策略

1. 在测试环境完成全量迁移与回归，采用脱敏或授权的生产副本验证性能。
2. 先发布后端兼容代码与数据库结构，再进行 GHCN 导入、聚合和数据核验，最后开启前端入口。
3. 用 `ADVANCED_ANALYSIS_ENABLED` 对内部用户灰度；监控稳定后逐步扩大。
4. 每次发布保留可回滚的应用版本、数据库迁移回退/修复脚本、数据导入水位线和导出文件兼容策略。
5. 任何阶段发现数据口径错误、越权、统计结果不可信或旧功能回归，立即关闭功能开关，保留证据并修复后重新验证。

## 附录：AI 执行时的工作纪律

1. 每修改一个模块，先阅读其现有实现和测试，再做最小兼容改动；不得用“重写整个模块”代替集成。
2. 每完成一个阶段，运行该阶段测试并报告：修改文件、接口变化、数据迁移、风险、验证结果和未决问题。
3. 对需求中未明确的业务口径（例如“中国东部”行政区边界、温度单位、预警通知渠道、租户模型），先从现有项目/数据字典查找；若仍无依据，记录假设并等待产品确认，不得私自把假设当成事实。
4. 对外部数据、统计结论和政策建议保留来源与方法；任何无法由数据验证的结论都要标记为推断。
