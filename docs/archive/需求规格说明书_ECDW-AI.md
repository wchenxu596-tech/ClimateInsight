# 2. 项目概述

## 2.1 项目背景

某大型电商平台日均产生数千万级用户行为数据（浏览、加购、收藏、购买）。传统BI报表开发存在以下痛点：

1. **开发周期长**：业务部门从提需求到拿到报表需要数天
2. **灵活性差**：每新增一个分析维度都需要重新开发
3. **数据利用率低**：海量行为数据仅做基本统计，未挖掘深层价值
4. **依赖人工**：报表撰写依赖数据分析师手工整理，无法自动化

## 2.2 项目目标

构建一套 **「数据仓库 + 智能分析 + 可视化大屏 + AI问答 + 自动报表」** 五位一体的电商分析平台，将数据分析周期从**天级缩短至分钟级**。

## 2.3 核心指标

| 指标类别 | 具体指标 | 目标值 |
|----------|----------|--------|
| 数据处理能力 | 支持的数据量级 | ≥1亿条用户行为记录 |
| 数据准确性 | ETL过程数据丢失/重复率 | 0% |
| 数仓分层 | 标准分层架构 | ODS → DWD → DWS → ADS 四层 |
| 分析模型 | 用户价值分群准确度 | 基于标准RFM算法，分群逻辑无误 |
| 查询性能 | ADS层查询响应时间 | < 3 秒 |
| NL2SQL准确率 | 自然语言→SQL正确率 | ≥ 85% |
| 报表自动化 | 从触发到生成报告 | < 5 分钟 |
| 可视化页面 | 分析页面数量 | ≥ 5 个独立页面 |
| 页面加载 | 前端首屏加载时间 | < 3 秒 |

## 2.4 项目范围

### 包含范围

- ✅ 淘宝用户行为数据集（1亿+记录）的导入与清洗
- ✅ Hive数据仓库四层架构设计与ETL
- ✅ RFM用户价值分群模型
- ✅ 购物篮关联规则分析
- ✅ Vue3 + ECharts 可视化大屏（5个以上页面）
- ✅ 自然语言→SQL智能问答系统
- ✅ LLM驱动的智能报表自动生成
- ✅ Flask后端API服务
- ✅ MySQL元数据与结果存储

### 不包含范围

- ❌ 实时流处理（本项目聚焦离线批处理分析）
- ❌ 用户登录/权限系统
- ❌ 移动端适配
- ❌ 生产级CI/CD流水线
- ❌ 实际线上电商系统对接

| **商品运营** | 负责选品、定价与促销策略 | 商品热度排行、关联推荐、品类分析 |

## 3.2 用户场景

### 场景1：运营经理的早晨
> 早上9点，运营经理打开总览大屏，查看昨日核心KPI（日活、订单量、GMV、转化率），发现转化率环比下降0.5%，随后进入用户分析页查看各分层用户占比变化，定位到「一般发展客户」群流失严重。

### 场景2：数据分析师的探索
> 分析师想了解"最近3天购买转化率最高的前10个商品类目"，在BI问答页输入自然语言问题，系统自动生成SQL并返回可视化图表，整个过程不到30秒。

### 场景3：商品运营的选品决策
> 商品运营在关联分析页查看购物篮规则，发现"购买了类目A的用户有42%同时购买了类目B"，据此调整组合推荐策略。

### 场景4：高层管理者的周报
> 每周一早8点，系统自动生成上周数据分析周报（含图表+文字洞察），发送至管理者的邮箱/页面。

## M1: 数据接入与ODS层管理 [P0]

### FR1.1 数据文件导入

**描述**：系统应支持将淘宝用户行为CSV文件导入HDFS，并在Hive ODS层建立对应的外部表。

**输入**：
- 文件名：`UserBehavior.csv`
- 格式：CSV，逗号分隔，无表头
- 字段：`user_id`, `item_id`, `category_id`, `behavior_type`, `timestamp`
- 编码：UTF-8

**处理规则**：
1. 文件上传至HDFS路径：`/raw_data/user_behavior/`
2. 按日期（`dt`）自动分区存储
3. 数据导入过程中不得修改任何原始字段值
4. 导入完成后记录日志：文件名、行数、导入时间、是否成功

**输出**：
- Hive外部表：`ods_user_behavior`
- 存储格式：ORC（优化行列存储）
- 分区字段：`dt`（日期，格式 yyyy-MM-dd）

**异常处理**：
- 文件格式不符时，返回明确错误信息
- 部分数据导入失败时，支持回滚或跳过异常行（记录到错误日志）
- 重复导入同一日期数据时，提示用户确认是否覆盖

### FR1.2 ODS层数据校验

**描述**：导入完成后自动执行数据校验。

**校验规则**：
| 校验项 | 规则 | 失败处理 |
|--------|------|----------|
| 行数校验 | CSV行数 = ODS表行数（按dt分区） | 告警，允许人工确认后继续 |
| 空值校验 | user_id / item_id / timestamp 不可为空 | 标记异常行 |
| 行为类型校验 | behavior_type ∈ {pv, cart, fav, buy} | 标记异常行 |
| 时间戳范围 | 时间戳在 2017-11-25 ~ 2017-12-03 之间 | 标记异常行 |
| 分区完整性 | 每晚一天都有对应分区 | 报告缺失分区 |

## M2: 数据仓库分层ETL [P0]

### FR2.1 DWD层 — 数据明细层

**描述**：对ODS原始数据进行清洗、标准化、维度退化，生成明细事实表和维度表。

**需生成的表**：

| 表名 | 类型 | 说明 |
|------|------|------|
| `dwd_user_behavior_fact` | 事实表 | 清洗后的用户行为明细 |
| `dwd_dim_item` | 维度表 | 商品维度（SCD Type 2） |
| `dwd_dim_date` | 维度表 | 日期维度（预生成） |

**清洗规则（`dwd_user_behavior_fact`）**：

| 规则编号 | 规则描述 |
|----------|----------|
| R1 | 生成唯一行为ID：`md5(user_id + item_id + ts + behavior_type)` |
| R2 | 行为类型标准化：`pv`→`page_view`, `cart`→`add_cart`, `fav`→`add_fav`, `buy`→`purchase` |
| R3 | 时间戳→可读时间字段：`behavior_time`(TIMESTAMP), `behavior_date`(STRING), `behavior_hour`(INT) |
| R4 | 维度退化：增加 `behavior_weekday`(1-7), `is_weekend`(0/1) |
| R5 | 数据质量标记：`data_quality_flag` = `valid` / `invalid` |
| R6 | 空值行：`data_quality_flag` 标记为 `invalid`，但保留在DWD中 |
| R7 | 异常行为类型：标记为 `invalid`，保留原始值以便追溯 |

**ETL执行规则**：
- 按天分区执行，支持增量更新
- 每个分区ETL完成后记录处理行数
- 若某分区ETL失败，不影响其他分区

### FR2.2 DWS层 — 数据汇总层 [P0]

**描述**：基于DWD明细数据，按主题进行轻度汇总。

**需生成的表**：

| 表名 | 粒度 | 说明 |
|------|------|------|
| `dws_user_behavior_day` | 用户+日期 | 用户行为日汇总 |
| `dws_item_behavior_day` | 商品+日期 | 商品行为日汇总 |

**`dws_user_behavior_day` 字段要求**：

| 字段 | 类型 | 计算逻辑 |
|------|------|----------|
| `pv_count` | BIGINT | COUNT(behavior_type='page_view') |
| `pv_items_distinct` | BIGINT | COUNT(DISTINCT item_id WHERE behavior='page_view') |
| `cart_count` | BIGINT | COUNT(behavior_type='add_cart') |
| `fav_count` | BIGINT | COUNT(behavior_type='add_fav') |
| `purchase_count` | BIGINT | COUNT(behavior_type='purchase') |
| `cart_rate` | DECIMAL(5,4) | cart_count / pv_count（除零返回NULL） |
| `purchase_rate` | DECIMAL(5,4) | purchase_count / pv_count（除零返回NULL） |
| `session_count` | BIGINT | 按30分钟无操作切分会话数 |
| `avg_session_duration` | INT | 平均会话时长（秒） |

**`dws_item_behavior_day` 字段要求**：

| 字段 | 类型 | 计算逻辑 |
|------|------|----------|
| `pv_count` / `cart_count` / `fav_count` / `purchase_count` | BIGINT | 同DWS用户表 |
| `pv_users_distinct` | BIGINT | COUNT(DISTINCT user_id WHERE behavior='page_view') |
| `purchase_users_distinct` | BIGINT | COUNT(DISTINCT user_id WHERE behavior='purchase') |
| `heat_score` | DECIMAL(10,2) | 0.4×pv + 0.2×cart + 0.2×fav + 0.2×purchase |

### FR2.3 ADS层 — 应用数据层 [P0]

**描述**：面向具体业务场景，生成可直接被前端消费的应用数据表。

**需生成的表**：

| 表名 | 用途 | 计算频率 |
|------|------|----------|
| `ads_user_rfm_segment` | RFM用户价值分层 | 每日更新 |
| `ads_market_basket_rules` | 购物篮关联规则 | 每日更新 |
| `ads_dashboard_kpi` | 大屏核心KPI | 每日更新 |

**`ads_user_rfm_segment` 要求**：

| 字段 | 说明 |
|------|------|
| `recency_days` | 最近一次购买距统计日天数 |
| `frequency` | 近90天购买天数 |
| `monetary` | 近90天购买金额（模拟） |
| `r_score` / `f_score` / `m_score` | 按NTILE(5)分位打分，1~5 |
| `rfm_score` | 拼接字符串，如"555" |
| `user_segment` | 中文分层标签（见FR3.1） |

**`ads_market_basket_rules` 要求**：

| 字段 | 说明 |
|------|------|
| `antecedent` / `consequent` | 前项/后项商品或类目ID |
| `support` | 支持度：A∩B占比 |
| `confidence` | 置信度：P(B\|A) |
| `lift` | 提升度：P(B\|A) / P(B) |
| `rule_desc` | 规则描述文本 |

**`ads_dashboard_kpi` 要求**：

至少包含以下KPI：
- 日活用户数（DAU）、日订单量、日GMV（模拟）、整体转化率
- 环比变化值（较前一日）
- 趋势标记（↑ / ↓ / →）

## M3: 分析模型引擎 [P1]

### FR3.1 RFM用户价值分层 [P1]

**描述**：基于用户购买行为进行RFM建模，将用户分为8个层级。

**分层规则**：

| R | F | M | 分层标签 |
|---|----|----|----------|
| ≥4 | ≥4 | ≥4 | 重要价值客户 |
| ≥4 | ≥4 | <4 | 重要保持客户 |
| ≥4 | <4 | ≥4 | 重要发展客户 |
| ≥4 | <4 | <4 | 重要挽留客户 |
| <4 | ≥4 | ≥4 | 一般价值客户 |
| <4 | ≥4 | <4 | 一般保持客户 |
| <4 | <4 | ≥4 | 一般发展客户 |
| <4 | <4 | <4 | 流失客户 |

**输出要求**：
- 每个用户有唯一的分层标签
- 支持按分层统计用户数量和占比
- 支持按分层查看RFM均值

### FR3.2 购物篮关联规则挖掘 [P1]

**描述**：基于用户购买共现关系，挖掘商品/类目之间的关联规则。

**算法要求**：
- 使用Apriori算法（至少支持2-项集）
- 最小支持度：可配置，默认 0.01
- 最小置信度：可配置，默认 0.3

**输出要求**：
- Top 50 关联规则，按提升度降序
- 每条规则包含：前项、后项、支持度、置信度、提升度
- 支持按支持度/置信度/提升度排序

## M5: NL2SQL智能问答引擎 [P1]

### FR5.1 自然语言理解

**描述**：将用户输入的自然语言问题解析为结构化查询意图。

**能力要求**：

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 日期解析 | 理解"昨天/最近7天/上周/本月"并转换为具体日期 | P0 |
| 指标识别 | "销售额"→`purchase_amount`，"用户数"→`COUNT(DISTINCT user_id)` | P0 |
| 维度识别 | "按类目"→`GROUP BY category_id`，"前10"→`LIMIT 10` | P0 |
| 聚合理解 | "总计/平均/最高/最低"→对应的聚合函数 | P0 |
| 排序理解 | "最多的/最高的/排名"→`ORDER BY ... DESC` | P0 |
| 过滤理解 | "大于/小于/等于/在...之间"→WHERE条件 | P1 |
| 多表关联 | 跨表查询时自动识别JOIN关系 | P2 |

### FR5.2 SQL生成

**描述**：基于解析结果，组装合法的Hive SQL。

**生成规则**：
1. 仅允许 SELECT 语句
2. 必须包含分区过滤条件（`WHERE dt = '...'`）
3. 表名/字段名严格来自预设的Schema白名单
4. 使用参数化方式防止注入

### FR5.3 SQL安全校验

**描述**：在SQL发送至Hive执行前进行安全检查。

**检查规则**：
- 禁止关键词：`DROP`, `DELETE`, `UPDATE`, `INSERT`, `TRUNCATE`, `ALTER`, `CREATE`
- 必须包含 `dt` 过滤条件（防止全表扫描）
- SQL长度不超过1000字符
- 仅允许访问白名单内的表和字段

**失败处理**：
- 包含危险操作 → 拒绝执行，提示"不支持修改数据操作"
- 缺少日期过滤 → 自动补充最近日期
- SQL超长 → 提示"请简化问题"

### FR5.4 结果可视化

**描述**：根据查询结果自动选择合适的图表类型。

**自动匹配规则**：

| 结果特征 | 图表类型 |
|----------|----------|
| 1个维度列 + 1个数值列，维度数≤10 | 柱状图 |
| 1个维度列 + 1个数值列，维度数>10 | 横向柱状图 |
| 1个时间列 + 1个数值列 | 折线图 |
| 1个类别列 + 1个数值列（占比） | 饼图/环形图 |
| 2个数值列 | 散点图 |
| 多个数值列 | 表格 |
| 单值结果 | 大数字卡片 |

### FR5.5 LLM集成

**描述**：调用大模型API实现问题理解和SQL生成。

**接口要求**：
- 支持 OpenAI API 兼容格式（可适配DeepSeek/通义千问等）
- API Key通过环境变量/配置文件注入
- 请求超时时间：30秒
- 失败重试：最多2次

## 6.2 数据字典

### 原始数据字段

| 字段 | 类型 | 说明 | 示例 | 是否可空 |
|------|------|------|------|----------|
| user_id | BIGINT | 用户ID（脱敏） | 1001 | 否 |
| item_id | BIGINT | 商品ID（脱敏） | 3826105 | 否 |
| category_id | BIGINT | 商品类目ID（脱敏） | 4076803 | 否 |
| behavior_type | STRING | 行为类型 | pv / cart / fav / buy | 否 |
| timestamp | BIGINT | Unix时间戳（秒） | 1511534400 | 否 |

### 行为类型枚举

| 值 | 含义 | 说明 |
|----|------|------|
| pv | page view | 页面浏览/点击 |
| cart | add to cart | 加入购物车 |
| fav | add to favorites | 添加收藏 |
| buy | purchase | 购买 |

> **注意**：这是一个用户行为日志数据集，不是交易数据集。数据中**不存在订单金额字段**。方案文档中的 `purchase_amount` 字段为**模拟生成**，需通过以下方式之一获得：
> - 方案A：随机生成合理范围的金额（如 9.9 ~ 9999 元），伪造成本低
> - 方案B：从类目ID哈希映射固定价格表（如 `category_id % 1000` 作为价格档位），可复现
> - 方案C：不引入金额维度，RFM中用购买次数替代M值，GMV等指标改为"订单量"

## 6.3 数据流图

```
[天池CSV] ──导入──→ [HDFS /raw_data/]

                  [ADS: ads_market_basket_rules]   ← 关联规则
                  [ADS: ads_dashboard_kpi]         ← 大屏KPI
                       │
                       ▼ (同步)
                  [MySQL: dw_model.*]              ← 前端查询
                       │
                       ▼
```
