# ClimateInsight 最终交付实施操作手册

> 适用范围：当前仓库的 2024 年 NOAA GSOD 气候数据 BI 项目。  
> 最终项目名称：**ClimateInsight：基于 NOAA GSOD 数据的全球气候智能分析平台**。  
> 目标：完成后可在新环境启动，导入 2024 数据，展示真实 BI 看板，并提供安全的受控分析 Agent。

---

## 0. 实施规则

1. **按本文顺序执行**。不要先做 Docker 或 Agent，再回头修数据库。
2. 当前只有 2024 数据。页面、README 和答辩材料统一写“2024 年全球气候分析”，不得声称已完成多年趋势或预警。
3. 统一数据链路：`NOAA CSV -> Hive ODS/DWD -> Python ETL -> MySQL DWS/ADS -> Flask -> Vue`。
4. Hive 的 `03_dws_etl.sql` 和 `04_ads_etl.sql` 不再作为运行链路；DWS/ADS 只由 Python 写入 MySQL。
5. 修改前执行：

   ```powershell
   git status
   git switch -c feature/final-delivery
   ```

   若项目尚未正确初始化 Git，先完成 Git 初始化和首次提交，再开始修改。

## 1. 修复 MySQL Schema 与 ETL 不匹配（必须最先做）

### 1.1 问题

`scripts/etl_to_mysql.py` 向 `dws_station_monthly` 写入以下字段：

```text
heat_wave_days, cold_wave_days, frost_days, snow_days, thunder_days
```

但 `sql/mysql/001_schema.sql` 未创建这些列，ETL 一执行就会报“Unknown column”。

### 1.2 修改 Schema

打开 `sql/mysql/001_schema.sql`，在 `dws_station_monthly` 的 `extreme_days` 后增加：

```sql
heat_wave_days INT DEFAULT 0,
cold_wave_days INT DEFAULT 0,
frost_days     INT DEFAULT 0,
snow_days      INT DEFAULT 0,
thunder_days   INT DEFAULT 0,
```

同时把所有 ADS 表调整为可存储多个年份：

```sql
-- ads_kpi：删除 kpi_name 单列主键，使用复合主键
PRIMARY KEY (kpi_name, data_year)

-- ads_ranking：增加唯一索引
UNIQUE KEY uk_ranking (category, data_year, rank_num)

-- ads_monthly_trend：不要再只用 obs_month 做主键
PRIMARY KEY (data_year, obs_month)

-- ads_zones：不要再只用 climate_zone 做主键
PRIMARY KEY (data_year, climate_zone)
```

### 1.3 将建库与升级拆分

不要在正式迁移中保留 `DROP TABLE IF EXISTS`。创建：

```text
sql/mysql/001_init.sql
sql/mysql/002_add_dws_weather_days.sql
sql/mysql/003_ads_year_keys.sql
```

建议内容：

- `001_init.sql`：只在空库创建表。
- `002_add_dws_weather_days.sql`：使用 `ALTER TABLE ... ADD COLUMN` 增加五个天气日字段。
- `003_ads_year_keys.sql`：备份后重建 ADS 表主键和唯一键。

如果数据库仅有演示数据，可以先备份并重建：

```powershell
mysqldump -u root -p climate_dw > backup_climate_dw.sql
mysql -u root -p < sql/mysql/001_init.sql
```

若数据库已有重要数据，必须使用增量 `ALTER TABLE`，不能直接 drop 表。

### 1.4 修改并验证 ETL

确认 `scripts/etl_to_mysql.py` 写入列与新 Schema 的列顺序完全一致。执行：

```powershell
$env:MYSQL_PASSWORD='你的密码'
python -B -m py_compile scripts/etl_to_mysql.py
python scripts/etl_to_mysql.py --year 2024
```

使用 MySQL 验证：

```sql
DESCRIBE climate_dw.dws_station_monthly;
SELECT COUNT(*) FROM climate_dw.dws_station_monthly WHERE year = 2024;
SELECT COUNT(*)
FROM climate_dw.dws_station_monthly
WHERE year = 2024
  AND (avg_temp NOT BETWEEN -90 AND 60 OR max_temp NOT BETWEEN -90 AND 60);
```

验收：ETL 无“Unknown column”；第二条异常温度查询返回 0；DWS 有 2024 数据。

## 2. 更新数据校验脚本

### 2.1 修改 `scripts/verify_data.py`

将硬编码的：

```python
DB = {"host":"127.0.0.1", "user":"root", "password":"123456", ...}
```

改为从环境变量读取，与 ETL 共用配置：

```python
DB = {
    "host": os.getenv("MYSQL_HOST", "127.0.0.1"),
    "port": int(os.getenv("MYSQL_PORT", "3306")),
    "user": os.getenv("MYSQL_USER", "climate_app"),
    "password": os.environ["MYSQL_PASSWORD"],
    "database": os.getenv("MYSQL_DATABASE", "climate_dw"),
    "charset": "utf8mb4",
}
```

同样把 Hive 容器名和 JDBC 地址换成：

```python
HIVE_CONTAINER = os.getenv("HIVE_CONTAINER", "hive-server")
HIVE_JDBC_URL = os.getenv("HIVE_JDBC_URL", "jdbc:hive2://hive-server:10000")
```

### 2.2 更正检查 SQL

将旧表 `dws_monthly` 全部改为 `dws_station_monthly`，并附加年份条件。

删除“ODS 必须大于 9000 万行”的固定阈值。改为：

```python
station_count = 12160
days_in_year = 366
max_rows = station_count * days_in_year
check("ODS 行数合理", 0 < ods <= max_rows, f"{ods:,}/{max_rows:,}")
```

增加以下检查：

```sql
-- DWS 重复主键
SELECT station_id, year, obs_month, COUNT(*) AS cnt
FROM dws_station_monthly
GROUP BY station_id, year, obs_month
HAVING cnt > 1;

-- ADS 月度数据完整性
SELECT COUNT(*)
FROM ads_monthly_trend
WHERE data_year = 2024;

-- 排名数据完整性
SELECT category, COUNT(*)
FROM ads_ranking
WHERE data_year = 2024
GROUP BY category;
```

### 2.3 执行验收

```powershell
$env:MYSQL_PASSWORD='你的密码'
python scripts/verify_data.py --year 2024
```

验收：所有校验通过；校验脚本中不再出现 `123456`、`dws_monthly`、`90000000`。


根目录 `.gitignore` 必须包含：

```gitignore
.env
backend/.env
frontend/.env
```

### 3.2 加载本地 `.env`

在 `backend/requirements.txt` 增加并锁定版本：

```text
python-dotenv==1.0.1
gunicorn==23.0.0
```

在 `backend/config.py` 最前面添加：

```python
from dotenv import load_dotenv
load_dotenv()
```

将 `MYSQL_USER` 默认值改为 `climate_app`；生产环境不应使用 root。

### 3.3 创建应用数据库账户

在 MySQL 中执行：

```sql
CREATE USER IF NOT EXISTS 'climate_app'@'%' IDENTIFIED BY '请替换为强密码';
GRANT SELECT, INSERT, UPDATE, DELETE ON climate_dw.* TO 'climate_app'@'%';
FLUSH PRIVILEGES;
```

如以后接入 LLM SQL 功能，另建只读账户：

```sql
CREATE USER IF NOT EXISTS 'climate_reader'@'%' IDENTIFIED BY '请替换为强密码';
GRANT SELECT ON climate_dw.* TO 'climate_reader'@'%';
FLUSH PRIVILEGES;
```

### 3.4 修复静态文件路由冲突

当前 Flask 同时注册默认静态文件路由和 SPA fallback，可能出现 `/<path:filename>` 与 `/<path:path>` 冲突。

修改 `backend/app.py`：

```python
app = Flask(__name__, static_folder="../frontend/dist", static_url_path="/static")
```

然后只保留一个 SPA fallback：

```python
@app.route("/", defaults={"path": ""})
@app.route("/<path:path>")
def spa(path):
    full = os.path.join(app.static_folder, path)
    if path and os.path.isfile(full):
        return send_from_directory(app.static_folder, path)
    return send_from_directory(app.static_folder, "index.html")
```

### 3.5 API 参数检查

在 `backend/routes/agent.py` 中把年份强制转为整数并校验：

```python
year = request.json.get("year", DATA_YEAR)
if not isinstance(year, int) or year not in {2024}:
    return jsonify({"code": 400, "message": "仅支持 2024 年数据"}), 400
```

在 `backend/agent/service.py` 中，优先使用请求参数 `year`，并覆盖意图识别出的年份，避免用户问“2023 年”但实际查询 2024、文本却显示 2023：

```python
intent_info = detect(question)
intent_info["year"] = year
```

异常时返回 HTTP 500 或 503，不要把数据库错误包装成 `code: 0` 的成功响应。

## 4. 完成受控分析 Agent

### 4.1 当前定位

保留当前“规则识别 + 白名单工具 + 模板回答”方案。它是安全且可交付的 Agent MVP，不需要模型 API Key。

当前支持范围固定为：

```text
全球 KPI
月度平均气温
最热 / 最冷 / 降水最多 / 极端天气最多的站点
气候带分布
帮助
```

禁止支持任意 SQL、写入、删除、跨库查询和没有数据支撑的多年比较。

### 4.2 后端 Agent 检查清单

检查以下文件：

```text
backend/agent/intents.py
backend/agent/tools.py
backend/agent/service.py
backend/agent/responder.py
backend/routes/agent.py
```

要求：

1. `tools.py` 中所有 SQL 必须参数化，且仅能访问 ADS 表。
2. `VALID_YEARS = {2024}` 与实际数据范围一致。
3. `limit` 限制在 1–15。
4. 用户输入最大 300 字；空输入、非 JSON、非法年份均返回 400。
5. 不能把用户输入直接作为 SQL 或表名的一部分。
6. Agent 结果必须带数据年份和限制说明。
7. `AGENT_ENABLED` 未实际用于 LLM 前，不要在页面上宣称“大模型已接入”。

### 4.3 前端 Agent 检查清单

检查 `frontend/src/views/BIAgent.vue`：

1. 使用 `askAgent(question, 2024)`，不能再调用 `/api/nl2sql`。
2. 聊天消息仅使用插值 `{{ }}` 渲染，禁止 `v-html`。
3. 表格数据先转换为对象，再交给 `el-table`。
4. 图表仅渲染后端已定义的 `bar`、`line`、`pie` 三种类型。
5. 请求时禁用重复发送；失败时给出可见提示。
6. 快捷问题至少覆盖 KPI、月度、排名、气候带四类。

### 4.4 Agent 手动验收

启动后端和前端后，依次输入：

```text
2024年全球平均气温？
2024年最热的10个站点？
各月温度变化？
气候带分布？
删除所有数据
SELECT * FROM ads_kpi
```

验收：前四项返回真实表格或图表；后两项被拒绝，且没有数据库写入。

### 4.5 可选：后续接入 LLM

只在 Agent MVP 稳定后实施。LLM 只能做两件事：

1. 将用户问题转换为受限 JSON 意图，如：

   ```json
   {"intent":"ranking","category":"hottest","limit":10,"year":2024}
   ```

2. 基于白名单工具返回的数据撰写解释。

LLM 不得生成或执行 SQL。后端必须使用 JSON Schema 验证模型返回结果；验证失败时回退到规则识别。

## 5. 创建 Docker 化部署

### 5.1 新增文件

创建：

```text
docker-compose.yml
backend/Dockerfile
frontend/Dockerfile
frontend/nginx.conf
backend/.dockerignore
frontend/.dockerignore
docs/07-部署运行手册.md
```

### 5.2 `backend/Dockerfile`

实现要求：

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 5000
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "app:app"]
```

### 5.3 `frontend/Dockerfile`

实现要求：先 `npm ci && npm run build`，再把 `dist/` 拷贝到 Nginx。不得将 `node_modules` 或本机 `dist` 直接复制进镜像。

### 5.4 `frontend/nginx.conf`

必须包含：

```nginx
location / {
  try_files $uri $uri/ /index.html;
}

location /api/ {
  proxy_pass http://backend:5000/api/;
}
```

### 5.5 `docker-compose.yml`

至少包含服务：

```text
mysql
backend
frontend
```

关键要求：

- MySQL 使用命名卷。
- MySQL 挂载 `sql/mysql/001_init.sql` 到初始化目录。
- MySQL 配置 healthcheck。
- backend 等待 MySQL healthcheck。
- frontend 对外暴露 `8080:80`；backend 和 MySQL 默认不对外暴露。
- 所有密码来自 `.env`。

### 5.6 Docker 验收

```powershell

# 编辑 .env，替换 change-me
docker compose config
docker compose up -d --build
docker compose ps
curl http://localhost:8080/api/health
```

验收：三项服务正常；浏览器访问 `http://localhost:8080` 可看到前端；前端请求不依赖 `localhost:5000`。

## 6. 前端最终收口

### 6.1 构建配置

保留：

```javascript
baseURL: import.meta.env.VITE_API_BASE_URL || '/api'
```

开发环境 Vite proxy 指向 `http://localhost:5000`；Docker 生产环境由 Nginx 代理 `/api`。

### 6.2 页面检查

每个页面都必须具备：

- Loading 状态。
- 错误状态和“重试”按钮。
- 空数据状态。
- 数据年份和 NOAA GSOD 来源说明。
- 小屏布局可换行。

趋势页保持“2024 年月度气温变化”，不要再显示需要多年数据的同比/年度趋势。

排名页为不同类别显示正确单位：

```text
hottest / coldest：°C
rainiest：mm
most_extreme：天
```

删除未使用的 Vite 默认 `HelloWorld.vue`、默认图标和模板 README，替换为项目真实 README。

### 6.3 前端验收

```powershell
cd frontend
npm ci
npm run build
```

验收：构建成功；浏览器控制台没有未处理异常；断开 API 时页面提示清晰。
