> 目标读者：AI / 新开发者，仅凭本文即可理解整个项目的完整结构和运行方式。

## 1. 项目简介

**ClimateInsight** 是一个基于 NOAA GSOD 全球气象数据的 BI 分析平台。支持 2022/2023/2024 三年的全球气象数据查询、可视化大屏展示、AI 智能问答。

- **技术栈**：Python Flask（后端） + Vue3/ElementPlus/ECharts（前端） + MySQL（数据库）
- **数据来源**：NOAA Global Summary of the Day (GSOD) https://www.ncei.noaa.gov/data/global-summary-of-the-day/archive/
- **数据链路**：NOAA原始CSV → Python预处理 → Python ETL聚合 → MySQL DWS/ADS → Flask API → Vue3大屏

## 3. 数据库设计（MySQL: climate_dw）

### DWS 层 — dws_station_monthly（气象站月度汇总）

| 字段 | 类型 | 说明 |
|------|------|------|
| station_id | VARCHAR(20) | 气象站ID |
| station_name | VARCHAR(200) | 气象站名称 |
| year | INT | 年份 |
| obs_month | INT | 月份(1-12) |
| latitude | DOUBLE | 纬度 |
| longitude | DOUBLE | 经度 |
| avg_temp | DOUBLE | 月均温(°C) |
| avg_temp_max | DOUBLE | 月均最高温 |
| avg_temp_min | DOUBLE | 月均最低温 |
| max_temp | DOUBLE | 月最高温 |
| min_temp | DOUBLE | 月最低温 |
| total_precip | DOUBLE | 月总降水(mm) |
| rainy_days | INT | 降雨天数 |
| extreme_days | INT | 极端天气天数 |
| heat_wave_days | INT | 热浪天数(≥35°C) |
| cold_wave_days | INT | 寒潮天数(≤-10°C) |
| frost_days | INT | 霜冻天数 |
| snow_days | INT | 降雪天数 |
| thunder_days | INT | 雷暴天数 |
| obs_days | INT | 观测天数 |
| climate_zone | VARCHAR(20) | 气候带(tropical/temperate/continental/polar/arid) |

主键：(station_id, year, obs_month)

### ADS 层 — 4张应用表

| 表名 | 主键 | 用途 |
|------|------|------|
| ads_kpi | (kpi_name, data_year) | 全球KPI：年均温、站点数、极端占比、最高温 |
| ads_monthly_trend | (data_year, obs_month) | 12个月全球均温/均最高/均最低 |
| ads_ranking | AUTO_INCREMENT id + UNIQUE(category, data_year, rank_num) | 4类排名各TOP15 |
| ads_zones | (data_year, climate_zone) | 5个气候带站点数 |

## 4. 后端架构

### 4.1 入口与配置

- `app.py`：创建Flask应用，注册5个Blueprint，配置CORS，提供SPA静态文件fallback
- `config.py`：所有配置从环境变量读取，通过 `python-dotenv` 加载 `.env` 文件
- `db.py`：pymysql连接封装。`query(sql, params)` 返回元组列表，`query_dict()` 返回字典列表。每次查询新建连接

### 4.2 API 端点（共8个）

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| GET | /api/health | — | 健康检查，返回DB状态 |
| GET | /api/kpi | year | 4项KPI指标 |
| GET | /api/monthly | year | 12个月温度数据 |
| GET | /api/zones | year | 5个气候带站点数 |
| GET | /api/ranking | year, category, limit | 站点排名（category: hottest/coldest/rainiest/most_extreme） |
| GET | /api/trend | year | 月度趋势（仅返回2024，同比需多年） |
| POST | /api/nl2sql | {question} | 占位端点，返回"功能待开放" |
| POST | /api/agent/query | {question, year} | AI分析助手 |

所有成功响应格式：`{"code":0, "message":"ok", "data":[...], "meta":{"data_year":2024}}`

### 4.3 AI Agent 架构

```
用户问题
    │
    ▼
agent.py（路由层）
    ├─ 输入校验：非JSON→400, 空问题→400, >300字→400, SQL关键词→400
    ├─ 年份校验：非整数→400, 不在{2022,2023,2024}→400
    │
    ▼
service.py（编排层）
    │
    ▼
intents.py（意图识别）
    ├─ detect_llm() → DeepSeek API（temperature=0, max_tokens=200）
    │   ├─ System Prompt 严格约束输出为10种JSON格式之一
    │   ├─ 解析JSON → Schema校验（字段完整性+类型检查）
    │   ├─ 校验失败 → 返回None
    │   └─ 超时5秒 → 返回None
    │
    └─ detect_rules() → 规则引擎（关键词+正则，兜底方案）
        └─ 支持的10种意图：kpi/monthly/ranking(4种)/zones/help/chat/unknown
    │
    ▼
tools.py（工具白名单）
    ├─ get_kpi(year)      → SELECT ads_kpi WHERE data_year=%s
    ├─ get_monthly(year)  → SELECT ads_monthly_trend WHERE data_year=%s
    ├─ get_ranking(y,c,l) → SELECT ads_ranking WHERE data_year=%s AND category=%s LIMIT %s
    └─ get_zones(year)    → SELECT ads_zones WHERE data_year=%s
    │
    ▼
responder.py（模板回答）
    ├─ kpi → 文本 + 表格
    ├─ monthly → 文本 + line图（支持月份范围过滤）
    ├─ ranking → 文本 + 表格 + bar图（站名自动翻译中文）
    ├─ zones → 文本 + 表格 + pie图
    ├─ chat → 自我介绍
    └─ help → 网站使用指南
```

**安全设计**：
- LLM只输出JSON意图，不生成SQL/代码/回答
- 工具层仅访问ADS表，全部参数化查询
- JSON Schema校验拒绝非法字段
- 任何环节失败都回退到规则引擎

|------|--------|------|

## 5. 前端架构

### 5.1 技术栈

- **Vue 3** (Composition API + `<script setup>`)
- **Element Plus** (UI组件库：表格、输入框、标签、弹窗)
- **ECharts** (图表：折线图、柱状图、饼图，按需加载渲染器)
- **Vue Router** (4条路由，HTML5 History模式)
- **Axios** (HTTP请求，baseURL默认 `/api`，15秒超时)

### 5.2 页面路由

| 路径 | 组件 | 功能 |
|------|------|------|
| / | Dashboard.vue | 总览：4个KPI卡片 + 月度温度折线 + 气候带饼图 + 最热TOP15柱状图 |
| /trend | TrendAnalysis.vue | 趋势：三条折线（均温/均最高/均最低）+ 12个月 |
| /ranking | CityRanking.vue | 排名：4类Radio切换 + 横向柱状图（带单位°C/mm/天） |
| /zones | ClimateZones.vue | 气候带：环形饼图 + 地图占位区 |
| 404 | redirect:/ | 所有未匹配路由重定向到首页 |

### 5.3 全局组件

- **年份选择器**：`App.vue` 右上角 `el-select`，通过 Vue `provide/inject` 传递给所有页面
- **AI助手**：`BIAgent.vue` 全局挂载（不在路由内），右下角悬浮🤖按钮 → 可拖拽可缩放的聊天面板

### 5.4 数据流

```
App.vue provide('selectedYear', ref(2024))
    │
    ├─ Dashboard.vue    inject → watch → load(year)
    ├─ TrendAnalysis    inject → watch → load(year)
    ├─ CityRanking      inject → watch → load(year)
    ├─ ClimateZones     inject → watch → load(year)
    └─ BIAgent.vue      inject → askAgent(q, year)
```

每个页面有完整的三态处理：**Loading**（骨架屏）→ **Error**（提示+重试按钮）→ **Empty**（空数据提示）。

### 5.5 关键工具函数

- `api/index.js`：`getKPI(y)`, `getMonthly(y)`, `getZones(y)`, `getRanking(y,cat,limit)`, `askAgent(q,y)`
- `utils/stationNames.js`：60+个英文站名 → 中文映射，`stationCN(name)` 函数
- ECharts tooltip 全局 `valueFormatter`：保留两位小数

## 6. 数据导入流程

### 完整流程（每个年份）

```bash
# 步骤1：下载NOAA原始数据
curl -L -o data/gsod_2022.tar.gz "https://www.ncei.noaa.gov/data/global-summary-of-the-day/archive/2022.tar.gz"

# 步骤2：解压
tar xzf data/gsod_2022.tar.gz -C data/gsod_2022/

# 步骤3：预处理（°F→°C，英寸→mm，天气标志解析，质量标记）
python scripts/preprocess_climate.py --input data/gsod_2022 --output data/climate_2022.csv --year 2022

# 步骤4：导入MySQL
export MYSQL_PASSWORD=123456
python scripts/etl_csv_to_mysql.py --year 2022 --csv data/climate_2022.csv
```

### 聚合逻辑（etl_csv_to_mysql.py 核心）

1. **读取预处理CSV** → 按 (station_id, year, month) 分组
2. **DWS聚合**：
   - 平均温度、最高/最低温
   - 降水量求和、降雨天数
   - 热浪(≥35°C)、寒潮(≤-10°C)、极端天气计数
   - 气候带分类（基于纬度+温度+降水规则）
   - 批量INSERT（500条/批）
3. **ADS计算**（从DWS）：
   - KPI：全球年均温、站点总数、极端占比、最高单站温度
   - 月度趋势：每月全球均温
   - 排名：4个类别各TOP15（按聚合值排序）
   - 气候带：每个zone的站点数

### 当前数据规模

| 年份 | 原始行数 | 站点数 | DWS记录 | 年均温 |
|------|----------|--------|---------|--------|
| 2022 | 3,994,699 | 12,315 | 138,333 | 13.1°C |
| 2023 | 4,038,747 | 12,308 | 139,870 | 13.7°C |
| 2024 | 3,931,419 | 12,154 | 137,928 | 14.0°C |

## 9. 关键设计决策

1. **CSV直写MySQL**：由于Hive/YARN不稳定，使用 `etl_csv_to_mysql.py` 直接从预处理CSV聚合写入MySQL，完全复刻原有ETL逻辑
2. **LLM作为分类器**：大模型只输出结构化JSON意图，不生成SQL或自然语言回答。回答由 `responder.py` 模板引擎生成
3. **三层安全防护**：API输入过滤 → JSON Schema校验 → 工具白名单（仅参数化查询ADS表）
4. **悬浮AI助手**：BIAgent从独立页面改为全局悬浮可拖拽面板，通过 `provide/inject` 获取当前年份
5. **年份选择器全局化**：通过Vue provide/inject机制传递，所有页面 `watch` 年份变化自动重新加载
6. **ADS复合主键**：`(kpi_name, data_year)`, `(data_year, obs_month)` 等支持多年数据共存
